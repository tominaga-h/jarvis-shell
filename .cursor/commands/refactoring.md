# /refactoring コマンド

ユーザーが指定したファイルやディレクトリに対して、可読性、パフォーマンス、安全性、およびアーキテクチャの一貫性を向上させるためのリファクタリングを提案・実行してください。機能の変更（振る舞いの変更）は行わず、内部構造の改善のみを目的とします。

## Instructions for AI

1. ユーザーが指定したコードを分析し、以下のルールに照らし合わせて改善点を洗い出してください。
2. 修正を行う前に、「どこを、なぜ、どのようにリファクタリングするか」の計画（Plan）を簡潔な箇条書きで提示してください。
3. ユーザーの承認（または指示）に基づき、コードの修正を適用してください。

リファクタリングを行う際は、以下の原則を厳守してください。

## 1. Idiomatic Rust（Rustらしいコード）

- `match` や `if let`、コンビネータ（`map`, `filter_map`, `and_then` など）を適切に使い、手続き型ではなく宣言的・関数型のアプローチを優先してください。
- 所有権と借用（Borrowing）のルールを最適化し、不要な `.clone()` や `.to_string()` を排除して、できる限り参照（`&str`, `&[T]`）を使用してください。
- 巨大な関数は、単一責任の原則（SRP）に従い、意味のある小さなヘルパー関数に分割してください。

## 2. Safety & Error Handling（安全性とエラーハンドリング）

- **Panicの徹底排除**: シェルというインフラツールの性質上、プロセスがクラッシュすることは絶対に許されません。`unwrap()` や `expect()` の使用は避け、`Result` と `?` 演算子を用いた適切なエラー伝播に書き換えてください。
- マルチバイト文字（日本語など）を扱う可能性がある文字列操作において、バイト単位でのスライス（例: `&s[..limit]`）によるパニックのリスクがないか点検し、`chars()` 等を用いた安全な文字境界の処理に修正してください。
- エラーが発生した場合は、原因を握りつぶさず（Silent Failureの防止）、ユーザーフレンドリーな警告（`stderr` への出力など）またはログ（`tracing::debug!`, `warn!` など）を残す構造にしてください。

## 3. Performance（パフォーマンスの最適化）

- 不要なヒープアロケーション（`Box`, `Vec`, `String` の無駄な生成）を避け、可能な限りスタック上のデータや参照を活用してください。
- ループ内での重い処理や、無駄なイテレータの `collect()` を見直し、遅延評価（Lazy Evaluation）を活用してください。

## 4. Architecture & Modularity（アーキテクチャと凝集度）

- Jarvishのレイヤー（`cli`, `engine`, `ai`, `storage`, `shell`）の境界を意識し、モジュール間の密結合を防いでください。
- 複数の場所で重複しているロジック（DRY原則違反）を見つけた場合は、適切な共通モジュール（ヘルパー）として抽出し、統合してください。

## 5. Documentation & Testing（ドキュメントとテスト）

- リファクタリングによって関数や構造体のシグネチャが変更された場合は、付随するRustdoc（`///`）も正確に更新してください。
- 既存のテストが壊れないようにリファクタリングを行うか、構造変更に合わせてテストコードも適切に修正してください。
