# v1.12.0 リリースノート

> コマンド実行中の `Ctrl+C` でシェルそのものが終了してしまう不具合を修正したリリースです。

## バグ修正

### コマンド実行中の `Ctrl+C` でシェルが落ちなくなりました — v1.12.0

これまで `sleep 100` のような長時間コマンドの実行中に `Ctrl+C` を押すと、実行中のコマンドだけでなく **jarvish 本体まで終了**してしまうケースがありました。本リリースで、`Ctrl+C` は実行中のコマンドだけを中断し、シェルはそのままプロンプトへ戻るようになりました ([#189](https://github.com/tominaga-h/jarvis-shell/issues/189))。

```bash
# 実行中に Ctrl+C → sleep だけ止まり、jarvish はそのまま継続
> sleep 100
^C
>                      # ← プロンプトに戻る（シェルは生きている）

# リダイレクトやパイプラインでも同じ挙動
> sleep 100 > /tmp/x
^C
> ping 8.8.8.8 | grep ttl
^C
```

#### 仕組み

一般的なシェルと同じ**ジョブ制御**を導入しました。外部コマンドを独立したプロセスグループに分離し（`setpgid`）、実行中は端末の前面プロセスグループをそのコマンドへ委譲（`tcsetpgrp`）、終了後に jarvish へ回収します。これにより `Ctrl+C` は実行中の子プロセスグループにのみ届き、jarvish には影響しません。

- リダイレクト付きコマンド・パイプライン・AI パイプ前段（`cmd | j "..."`）のレガシー / captured 経路でもシェルが落ちなくなりました
- `vim` / `less` などの対話プログラム（PTY 経路）の挙動は従来どおりです
- プロンプト入力中の `Ctrl+C`（行のクリア）、AI 応答ストリーム中の `Ctrl+C`（ストリーム中断）も従来どおり動作します
- 起動時に SIGINT を無視する保険は採用していません。AI ストリーム中断（SIGINT 受信ベース）や reedline の `Ctrl+C` 処理を壊さないため、ジョブ制御（`tcsetpgrp`）のみで対処しています

## 内部実装

- **`src/engine/job_control.rs` 新設**: ジョブ制御の純粋ロジックと薄い syscall ラッパを集約。`setpgid` はベストエフォート（失敗してもコマンドは続行）、`tcsetpgrp` は `SIGTTOU` を一時退避して呼び出し、端末は `TerminalForegroundGuard`（RAII）で確実に jarvish へ回収
- **`cfg!(test)` / 非 tty 時はジョブ制御をスキップ**: テスト並列実行時の端末競合を回避（既存 PTY 経路の判定と整合）
- **テスト追加**: ジョブ制御の適用可否判定とパイプライン pgid 決定ロジックを純粋関数として切り出し、ユニットテスト 8 件を追加

## 対応プラットフォーム

- macOS (aarch64 / x86_64)
- Linux (x86_64)

## 既知の制限

- Windows は未対応
- AI 機能の利用には OpenAI API キーが必要
