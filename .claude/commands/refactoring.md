指定したファイルやディレクトリに対して、可読性・パフォーマンス・安全性・アーキテクチャの一貫性を向上させるリファクタリングを提案・実行する。機能の変更（振る舞いの変更）は行わず、内部構造の改善のみを目的とする。

対象: `$ARGUMENTS`

## 手順

1. 指定されたコードを分析し、以下のルールに照らし合わせて改善点を洗い出す
2. 修正を行う前に、「どこを、なぜ、どのようにリファクタリングするか」の計画を簡潔な箇条書きで提示する
3. ユーザーの承認に基づき、コードの修正を適用する

## リファクタリング原則

### 1. Idiomatic Rust

- `match` や `if let`、コンビネータ（`map`, `filter_map`, `and_then` など）を適切に使い、宣言的・関数型のアプローチを優先する
- 不要な `.clone()` や `.to_string()` を排除し、できる限り参照（`&str`, `&[T]`）を使用する
- 巨大な関数は単一責任の原則（SRP）に従い、意味のある小さなヘルパー関数に分割する

### 2. Safety & Error Handling

- **Panicの徹底排除**: シェルというインフラツールの性質上、プロセスのクラッシュは絶対に許されない。`unwrap()` や `expect()` を避け、`Result` と `?` 演算子を用いた適切なエラー伝播に書き換える
- マルチバイト文字（日本語など）を扱う文字列操作でバイト単位スライスによるパニックのリスクがないか点検し、`chars()` 等を用いた安全な文字境界処理に修正する
- エラーは握りつぶさず（Silent Failure の防止）、`tracing::warn!` / `debug!` 等でログを残す

### 3. Performance

- 不要なヒープアロケーション（`Box`, `Vec`, `String` の無駄な生成）を避け、スタック上のデータや参照を活用する
- ループ内での重い処理や無駄な `collect()` を見直し、遅延評価（Lazy Evaluation）を活用する

### 4. Architecture & Modularity

- Jarvish のレイヤー（`cli`, `engine`, `ai`, `storage`, `shell`）の境界を意識し、モジュール間の密結合を防ぐ
- 重複しているロジック（DRY原則違反）は適切な共通モジュールとして抽出・統合する

### 5. Documentation & Testing

- 関数や構造体のシグネチャが変更された場合は付随する Rustdoc（`///`）も正確に更新する
- 既存のテストが壊れないようリファクタリングするか、構造変更に合わせてテストコードも修正する
